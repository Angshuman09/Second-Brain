
# ==**Hash Index(most important)**==

We already saw the database log like this:

```
123,London  
42,San Francisco  
42,Exploratorium
```

Problem:  
To find key `42`, we must **scan the whole file**.

Solution:

Use a **hash map as an index**.

Structure:

```
HashMap (in RAM)  
  
123 -> byte 0  
42 -> byte 40
```

Meaning:

- key → **location in the file**
    

That location is called:

### **Byte Offset**

Example:

```
42 -> byte 40
```
Means:

> Start reading the file at byte 40 to get the value.

So lookup becomes:

```
Step 1: find key in hash map  
Step 2: jump directly to file offset  
Step 3: read value
```

This makes lookup **O(1)** instead of **O(n)**.

![[HashIndex.png]]

# ==Real Database That Uses This Idea==

A real storage engine that works like this is:

- **Bitcask** used in Riak.
    

Architecture:

```
Disk  
 └── append-only log (values)  
  
RAM  
 └── hash map (key → file offset)
```

Important rule:

> **All keys must fit in RAM.**

Values can be huge because they stay on disk.

# ==Why This Is Very Fast==

Reads:

```
hash map lookup → jump to disk
```

Only **one disk seek**.

Sometimes even **0 disk reads** if data is cached.

Writes:

```
append to log  
update hash map
```

Both are very fast.

So this design gives:

|Operation|Speed|
|---|---|
|Write|Very fast|
|Read|Very fast|

# ==Real-World Use Case==

The book gives example:

```
cat video URL -> play count
```

Example:

```
youtube.com/cat123 -> 5
```

Each time someone clicks:

```
youtube.com/cat123 -> 6  
youtube.com/cat123 -> 7  
youtube.com/cat123 -> 8
```

Updates happen **many times per key**.

But number of keys is **small enough to fit in RAM**.

Perfect for **hash index design**.

---

# ==The Disk Space Problem==

Since we **never overwrite**, the file grows forever.

Example:

```
cat1 -> 1  
cat1 -> 2  
cat1 -> 3  
cat1 -> 4
```

Old values remain.

Solution:

### **Segment the log**

Instead of one huge file:

```
segment1.log  
segment2.log  
segment3.log
```

When file reaches size limit:

```
close segment  
create new segment
```

# ==Compaction== (VERY IMPORTANT)

Compaction removes old values.

Example log:
```
cat1 -> 1  
cat2 -> 4  
cat1 -> 2  
cat1 -> 3
```

After compaction:

```
cat1 -> 3  
cat2 -> 4
```

Only **latest values remain**.

>  Compaction means throwing away duplicate keys in the log, and keeping only the most recent update for each key.

# ==Segment Merging==

- Multiple **segments can be merged into one segment** to reduce the total number of segment files.
    
- Since segments are **immutable (never modified after writing)**:
    
    - The merged result is written to a **new segment file**.
        
    - After merging is complete, **old segments are deleted**.
        
- Merging runs in a **background thread**, so the database can still handle **reads and writes normally**.
    
- After merging finishes, **read operations switch to the new merged segment**.
    

![[SegmentMerging.png]]

---
# ==Lookup Process==

Each segment maintains an **in-memory hash table**:

```
key → file offset
```

To find a value:

1. Check the **most recent segment**.
    
2. If not found → check **2nd most recent segments**.
    
3. Continue until the key is found.
    

Segment merging helps **reduce the number of segments**, making lookups faster.

---
# ==Practical Implementation Concerns==

## File Format

- Text formats like CSV are inefficient.
    
- Databases prefer **binary formats**:
    
    ```
    [length of string] + [raw string]
    ```
    

## Deleting Records (Tombstones)

- To delete a key, a **special deletion record (tombstone)** is appended.
    
- During segment merging:
    
    - Old values for that key are **discarded**.
        

## Crash Recovery

- In-memory hash tables are **lost after restart**.
    
- They can be rebuilt by **scanning segment files**.
    
- Optimization: store **hash map snapshots on disk** to recover faster.
    

## Partially Written Records

- If a crash happens during a write, records may be incomplete.
    
- **Checksums** are used to detect corrupted records and ignore them.
    

## Concurrency Control

- Usually **one writer thread** writes sequentially to the log.
    
- Segments are **append-only**, allowing **multiple threads to read concurrently**.
    

---
# ==Why Append-Only Logs?==

Advantages:

- **Sequential writes** → faster than random writes.
    
- **Simpler crash recovery**.
    
- **Simpler concurrency control**.
    
- **Segment merging prevents file fragmentation**.

---

# ==Limitations of Hash-Table Indexing==

## 1. Index Must Fit in Memory

- The hash table storing `key → offset` must be **kept in RAM**.
    
- If the number of keys becomes too large, memory becomes a bottleneck.
    

---

## 2. Inefficient Range Queries

Hash tables do not preserve key ordering.

Example query:

```
keys between kitty00000 and kitty99999
```

Problem:

- The database must check **each key individually** instead of scanning a range efficiently.

---

# ==SSTables and LSM-Trees==

##  Problem with Simple Log Segments

Earlier design:

- Data stored as **append-only log**
    
- **Hash index in memory**
    

Issues:

- Large memory needed for index
    
- Merging segments inefficient
    

---

# ==SSTable (Sorted String Table)==

**Definition:**

SSTable = log segment where **key-value pairs are sorted by key**

Rules:

- Keys are **sorted**
    
- Each key appears **only once per segment** (after compaction)

![[SSTable.png]]

---

#  ==Advantages of SSTables==

## 1.  Efficient Merging

Segments can be merged **like merge sort**

Process:

1. Read multiple segment files
    
2. Compare first keys
    
3. Write **smallest key** to new file
    
4. Repeat
    

If **same key appears in multiple segments**:

- Keep **newest value**
    
- Remove **older values**
    

Reason:  
Newer segments contain **more recent writes**

---

## 2.  Smaller In-Memory Index

You **don't need index for every key**

Instead use **sparse index**

Example:  
Index stores only some keys:

```
handbag → offset
handsome → offset
```

If searching **handiwork**:

- Jump to **handbag offset**
    
- Scan forward until found
    

Benefits:

- Much **less memory**
    
- **Fast scanning**
    

![[SSTable2.png]]

---

## 3.  Compression

Because keys are sorted and scanned sequentially:

Data can be stored as **compressed blocks** (indicated by the shaded area in Figure 3-5)

Structure:

```
Index → start of compressed block
Block → many key-value pairs
```

Benefits:

- Less disk space
    
- Less disk I/O
    
- Faster reads
    

---
# ==Constructing and Maintaining SSTables==

## Problem

Writes arrive **in random order**, but SSTables require **keys sorted by key**.

Maintaining sorted structure **directly on disk is hard**, but **easy in memory**.

Solution: use a **balanced tree in memory**.

Examples:

- Red-Black Tree
    
- AVL Tree
    

These allow:

- Insert keys in any order
    
- Read keys in sorted order
    

---

# ==Storage Engine Workflow==

## 1. Write Path (Memtable)

When a write arrives:

1. Insert the key-value pair into an **in-memory balanced tree**
    
2. This tree is called **Memtable**
    

Memtable properties:

- Stored in memory
    
- Automatically keeps keys **sorted**
    

---

## 2. Flushing Memtable to Disk

When the **memtable becomes large** (few MB):

1. Write the memtable to disk as an **SSTable**
    
2. Since the tree is already sorted, writing is efficient
    
3. The SSTable becomes the **newest segment**
    

During this time:

- A **new memtable** is created
    
- Writes continue normally
    

---

## 3. Read Path

To read a key:

1. Check **memtable**
    
2. Check **newest SSTable**
    
3. Check **older SSTables sequentially**
    

Search order:

```
Memtable
↓
Newest SSTable
↓
Older SSTables
```

---

## 4. Compaction

Background process periodically:

- **Merge SSTables**
    
- Remove **duplicate keys**
    
- Keep **latest value**
    
- Remove **deleted records**
    

Purpose:

- Reduce number of files
    
- Improve read efficiency
    

---

# ==Crash Problem==

If database crashes:

- Data in **memtable is lost**
    
- Because it was **not written to disk yet**
    

---

# Solution: Write-Ahead Log (WAL)

Every write is also appended to a **separate disk log**.

Steps:

1. Write operation arrives
    
2. Append to **log on disk**
    
3. Insert into **memtable**
    

If crash happens:

- Recover memtable by **replaying the log**
    

When memtable is flushed to SSTable:

- Corresponding log entries can be **deleted**
    

---

# ==LSM-Tree (Log Structured Merge Tree)==

The architecture described above is called:

**LSM-Tree**

Key idea:

```
Writes → Memtable
Memtable full → SSTable
Background → Merge + Compaction
```

Storage engines using this approach:

- LevelDB
    
- RocksDB
    
- Cassandra
    
- HBase
    

---

# ==Related System==

**Lucene (used in Elasticsearch and Solr)** also uses a similar concept.

Example:

```
Key   → word (term)
Value → list of document IDs containing the word
```

These mappings are stored in **SSTable-like sorted files** and merged over time.

---
# ==Performance Optimizations (LSM-Trees)==

## 1. Problem: Slow Lookup for Non-Existing Keys

In LSM-trees, when searching for a key:

Search order:

```
Memtable → Newest SSTable → Older SSTables
```

If the key **does not exist**, the system must check **many SSTables**, causing multiple disk reads.

---
## 2. Solution: Bloom Filters

**Bloom Filter** = memory-efficient probabilistic data structure.

Purpose:

- Quickly check **if a key definitely does NOT exist**
    

Behavior:

```
Bloom Filter says "not present" → skip disk read  
Bloom Filter says "maybe present" → check SSTable
```

Benefit:

- Avoids many unnecessary disk reads.
    

---
## 3. Compaction Strategies

LSM-trees periodically **merge SSTables**. There are two main strategies.

### Size-Tiered Compaction

Idea:

- Merge **similar-sized SSTables**
    

Process:

```
Small SSTables → merged → larger SSTables
```

Used in:

- Cassandra
    
- HBase
    

Characteristics:

- Fewer compactions
    
- Higher write throughput
    
- More duplicate data temporarily
    

---

### Leveled Compaction

Idea:

- Data organized into **levels**
    

Structure:

```
Level 0 → newest data  
Level 1  
Level 2  
Level 3
```

Properties:

- Each level is larger than the previous
    
- Key ranges are split into smaller SSTables
    
- Data gradually moves to deeper levels
    

Used in:

- LevelDB
    
- RocksDB
    

Benefits:

- More predictable reads
    
- Less space amplification
    

---

## 4. Strengths of LSM-Trees

Key advantages:

1. Works well even when **dataset >> memory**
    
2. Supports **range queries** (data is sorted)
    
3. **Sequential disk writes**
    
4. Very **high write throughput**
    

Core idea:

```
Many SSTables  
+ 
Background merging
```
