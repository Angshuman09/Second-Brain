B-Trees are the **most widely used database index structure**.

Used in:

- Most relational databases
    
- Many NoSQL databases
    

Introduced in **1970**.

---

# ==Key Idea==

Like SSTables:

```
Keys are sorted
```

But the **design philosophy is different**.

---
# ==LSM-Tree vs B-Tree==

| Feature | LSM-Tree | B-Tree |
|--------|----------|--------|
| Storage Structure | Large sequential files | Fixed-size pages |
| Write Pattern | Append-only writes | In-place updates |
| Disk Access | Mostly sequential | Random reads and writes |
| Maintenance | Background compaction | Page splits and updates |

---

# ==B-Tree Structure==

B-tree stores data in **pages (blocks)**.

Typical page size:

```
4 KB
```

Pages contain:

```
keys + references to child pages
```

Pages form a **tree structure**.

![[B-tree.png]]

---
# ==Searching in a B-Tree==

Steps:

1. Start at **root page**
    
2. Compare key with boundaries
    
3. Follow the appropriate **child reference**
    
4. Continue until reaching a **leaf page**
    

Leaf page contains:

- key-value pairs
    
- or pointer to value
    

Search complexity:

O(log n)

---

# ==Branching Factor==

Branching factor = number of child references per page.


> The number of references to child pages in one page of the B-tree is called the
> branching factor. For example, in Figure 3-6 the branching factor is six. In practice,
> the branching factor depends on the amount of space required to store the page 
> references and the range boundaries, but typically it is several hundred.


Typical value:

```
Hundreds
```

Example:

```
branching factor = 500
```

Result:

- Tree depth very small
    
- Usually **3–4 levels**
    

Example capacity:

```
4 levels + branching factor 500  
≈ 256 TB data
```

---

# ==Updating Data in B-Tree==

### Update existing key

1. Find leaf page
    
2. Modify value
    
3. Write page back to disk
    
---
### Insert new key

1. Find appropriate leaf page
    
2. Insert key
    

If page **is full**:

split page → two half-full pages  
update parent page

This keeps the tree **balanced**.

# ==Making B-Trees Reliable==

## 1. How B-Trees Write Data

Core write operation:

```
Overwrite a page on disk
```

Important assumption:

- The **page location does not change**
    
- Other pages referencing it remain valid
    

Contrast with LSM-tree:

|B-Tree|LSM-Tree|
|---|---|
|Overwrites pages|Append-only writes|
|Modifies data in place|Never modifies files|

---

# 2. Problem: Crash During Updates

Some operations modify **multiple pages**.

Example: [[The page Split problem]]

Steps:

1. Split full page into **two pages**
    
2. Update **parent page**
    
3. Write all pages to disk
    

Problem:

```
Crash during these writes → inconsistent index
```

Example corruption:

- Orphan page
    
- Broken references
    

---

# 3. Solution: Write-Ahead Log (WAL)

To prevent corruption, B-trees use **Write-Ahead Log (WAL)**.

Definition:

```
Append-only log storing every modification
before applying it to the B-tree pages
```

Steps:

1. Write change to **WAL**
    
2. Apply change to **B-tree page**
    
3. Write page to disk
    

If crash happens:

```
Recover database using WAL
```

WAL is also called:

```
Redo Log
```

---

# 4. Concurrency Problem

Multiple threads may access the B-tree simultaneously.

Problem:

```
Thread may read tree while another thread modifies it
```

Result:

```
Inconsistent state
```

---

# 5. Solution: Latches (Locks)

B-tree implementations use **latches**.

Definition:

```
Lightweight locks protecting tree pages
```

Purpose:

- Prevent simultaneous conflicting modifications
    
- Maintain consistency
    

Comparison:

|B-Tree|LSM-Tree|
|---|---|
|Requires locking|Simpler concurrency|
|Updates pages in place|Background merges|

---

# ==B-Tree Optimizations==

## 1. Copy-on-Write

Some databases avoid overwriting pages.

Example:

```
LMDB
```

Method:

1. Write modified page to **new location**
    
2. Create new parent pages pointing to it
    

Benefits:

- Crash safety
    
- Better concurrency
    

---

## 2. Key Abbreviation

Instead of storing full keys:

```
Store shortened keys
```

Especially useful in **internal nodes**.

Benefit:

```
More keys per page → larger branching factor → fewer tree levels
```

---

## 3. Sequential Layout of Leaf Pages

Problem:

```
Pages may be scattered across disk
```

Range queries may cause many **disk seeks**.

Optimization:

```
Store leaf pages sequentially on disk
```

But this is **hard to maintain as tree grows**.

Comparison:

|Structure|Disk Layout|
|---|---|
|B-tree|Pages scattered|
|LSM-tree|Sequential segments|

---

## 4. Sibling Pointers

Leaf pages may contain pointers to:

```
Left sibling
Right sibling
```

Purpose:

```
Efficient range scanning
```

Without returning to parent nodes.

---

## 5. Fractal Trees

A B-tree variant.

Idea:

```
Borrow log-structured techniques
```

Goal:

```
Reduce disk seeks
```

---

# ==Core Idea to Remember==

**B-tree reliability is achieved using:**

```
Write-Ahead Log (WAL)
+
Locks (latches)
+
Page management techniques
```

# ==Comparing B-Trees and LSM-Trees==

## General Rule of Thumb

|Aspect|LSM-Tree|B-Tree|
|---|---|---|
|Write performance|Faster writes|Slower writes|
|Read performance|Slower reads|Faster reads|
|Reason|Must check multiple SSTables|Data located in one place|

Reason for slower reads in LSM-tree:

```
Memtable → Multiple SSTables → Different compaction stages
```

---

# ==Benchmark Reality==

Performance comparisons are **workload dependent**.

Meaning:

```
Different workloads → different results
```

Therefore:

```
Always benchmark using your real workload
```

---

# Advantages of LSM-Trees

## 1. Higher Write Throughput

B-tree writes data multiple times:

```
1. Write to WAL
2. Write to tree page
3. Page split updates
```

LSM-tree writes:

```
Sequential SSTable writes
```

Sequential disk writes are **much faster than random writes**, especially on HDDs.

---

## 2. Lower Write Amplification (Sometimes)

**Write amplification**

Definition:

```
One logical write → multiple physical disk writes
```

Example sources:

- WAL
    
- Page splits
    
- Compaction
    

Lower amplification → higher throughput.

Important for:

```
SSDs (limited write cycles)
```

---

## 3. Better Compression

LSM-trees compress data more effectively because:

```
Sorted data + block compression
```

B-trees waste space due to:

```
Fragmentation
Half-filled pages after splits
```

Result:

|Structure|Storage Efficiency|
|---|---|
|LSM-tree|Smaller disk usage|
|B-tree|More fragmentation|

---

## 4. Sequential Disk Layout

LSM-tree writes **large sequential files**.

Benefit:

```
Better disk bandwidth usage
```

Especially beneficial for:

```
Magnetic hard drives
```

---

# Downsides of LSM-Trees

## 1. Compaction Overhead

LSM-trees require **background compaction**.

Problem:

```
Compaction uses disk resources
```

Effects:

- Slower reads
    
- Slower writes
    

Sometimes causes:

```
High tail latency (slow worst-case queries)
```

B-trees often provide **more predictable latency**.

---

## 2. Compaction Falling Behind

If write rate is high:

```
Incoming writes > compaction speed
```

Then:

```
Unmerged SSTables increase
```

Consequences:

```
More disk usage
Slower reads
Possible disk exhaustion
```

Requires:

```
Monitoring and tuning
```

---

## 3. Multiple Copies of Keys

In LSM-trees:

```
Same key may exist in multiple SSTables
```

Because:

```
Older versions removed during compaction
```

In B-tree:

```
Each key stored in exactly one place
```

---

# ==Transactional Advantage of B-Trees==

Relational databases often use **range locks**.

Example:

```
Lock keys between A and B
```

This works naturally in B-trees because:

```
Keys stored in one ordered structure
```

LSM-trees make this harder.

---

# Practical Reality

|Property|LSM-Tree|B-Tree|
|---|---|---|
|Write-heavy workloads|Better|Worse|
|Read-heavy workloads|Worse|Better|
|Disk efficiency|Better|Worse|
|Latency predictability|Worse|Better|
|Transaction support|Harder|Easier|

---

# Key Takeaway

```
LSM-tree → optimized for writes
B-tree → optimized for reads and transactions
```

There is **no universally best choice**.

Correct approach:

```
Benchmark using your actual workload
```
