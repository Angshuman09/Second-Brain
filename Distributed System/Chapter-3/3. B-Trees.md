B-Trees are the **most widely used database index structure**.

Used in:

- Most relational databases
    
- Many NoSQL databases
    

Introduced in **1970**.

---

# ==Key Idea==

Like SSTables:

```
Keys are sorted
```

But the **design philosophy is different**.

---
# ==LSM-Tree vs B-Tree==

| Feature | LSM-Tree | B-Tree |
|--------|----------|--------|
| Storage Structure | Large sequential files | Fixed-size pages |
| Write Pattern | Append-only writes | In-place updates |
| Disk Access | Mostly sequential | Random reads and writes |
| Maintenance | Background compaction | Page splits and updates |

---

# ==B-Tree Structure==

B-tree stores data in **pages (blocks)**.

Typical page size:

```
4 KB
```

Pages contain:

```
keys + references to child pages
```

Pages form a **tree structure**.

![[B-tree.png]]

---
# ==Searching in a B-Tree==

Steps:

1. Start at **root page**
    
2. Compare key with boundaries
    
3. Follow the appropriate **child reference**
    
4. Continue until reaching a **leaf page**
    

Leaf page contains:

- key-value pairs
    
- or pointer to value
    

Search complexity:

O(log n)

---

# ==Branching Factor==

Branching factor = number of child references per page.


> The number of references to child pages in one page of the B-tree is called the
> branching factor. For example, in Figure 3-6 the branching factor is six. In practice,
> the branching factor depends on the amount of space required to store the page 
> references and the range boundaries, but typically it is several hundred.


Typical value(In general branching factor):

```
Hundreds
```

Example:

```
branching factor = 500
```

Result:

- Tree depth very small
    
- Usually **3–4 levels**
    

Example capacity:

```
4 levels + branching factor 500  
≈ 256 TB data
```

---

# ==Updating Data in B-Tree==

### Update existing key

1. Find leaf page
    
2. Modify value
    
3. Write page back to disk
    
---
### Insert new key

1. Find appropriate leaf page
    
2. Insert key
    

If page **is full**:

split page → two half-full pages  
update parent page

This keeps the tree **balanced**.

![[B-tree2.png]]

# ==Making B-Trees Reliable==

## 1. How B-Trees Write Data

Core write operation:

[[Overwrite a page on disk]]


Important assumption:

- The **page location does not change**
    
- Other pages referencing it remain valid
    

Contrast with LSM-tree:

|B-Tree|LSM-Tree|
|---|---|
|Overwrites pages|Append-only writes|
|Modifies data in place|Never modifies files|

---

# 2. Problem: Crash During Updates

Some operations modify **multiple pages**.

Example: [[The page Split problem]]

Steps:

1. Split full page into **two pages**
    
2. Update **parent page**
    
3. Write all pages to disk
    

Problem:

```
Crash during these writes → inconsistent index
```

Example corruption:

- Orphan page
    
- Broken references
    

---

# 3. Solution: Write-Ahead Log (WAL)

To prevent corruption, B-trees use **Write-Ahead Log (WAL)**.

Definition:

```
Append-only log storing every modification
before applying it to the B-tree pages
```

Steps:

1. Write change to **WAL**
    
2. Apply change to **B-tree page**
    
3. Write page to disk
    

If crash happens:

```
Recover database using WAL
```

WAL is also called:

```
Redo Log
```

---

# 4. Concurrency Problem

Multiple threads may access the B-tree simultaneously.

Problem:

```
Thread may read tree while another thread modifies it
```

Result:

```
Inconsistent state
```

---

# 5. Solution: Latches (Locks)

B-tree implementations use **latches**.

Definition:

```
Lightweight locks protecting tree pages
```

Purpose:

- Prevent simultaneous conflicting modifications
    
- Maintain consistency
    

Comparison:

|B-Tree|LSM-Tree|
|---|---|
|Requires locking|Simpler concurrency|
|Updates pages in place|Background merges|

---

# ==B-Tree Optimizations==

## 1. Copy-on-Write

Some databases avoid overwriting pages.

Example:

```
LMDB
```

Method:

1. Write modified page to **new location**
    
2. Create new parent pages pointing to it
    

Benefits:

- Crash safety
    
- Better concurrency
    

---

## 2. Key Abbreviation

Instead of storing full keys:

```
Store shortened keys
```

Especially useful in **internal nodes**.

Benefit:

```
More keys per page → larger branching factor → fewer tree levels
```

---

## 3. Sequential Layout of Leaf Pages

Problem:

```
Pages may be scattered across disk
```

Range queries may cause many **disk seeks**.

Optimization:

```
Store leaf pages sequentially on disk
```

But this is **hard to maintain as tree grows**.

Comparison:

|Structure|Disk Layout|
|---|---|
|B-tree|Pages scattered|
|LSM-tree|Sequential segments|

---

## 4. Sibling Pointers

Leaf pages may contain pointers to:

```
Left sibling
Right sibling
```

Purpose:

```
Efficient range scanning
```

Without returning to parent nodes.

---

## 5. Fractal Trees

A B-tree variant.

Idea:

```
Borrow log-structured techniques
```

Goal:

```
Reduce disk seeks
```

---

# ==Core Idea to Remember==

**B-tree reliability is achieved using:**

```
Write-Ahead Log (WAL)
+
Locks (latches)
+
Page management techniques
```

# ==Comparing B-Trees and LSM-Trees==

## General Rule of Thumb

|Aspect|LSM-Tree|B-Tree|
|---|---|---|
|Write performance|Faster writes|Slower writes|
|Read performance|Slower reads|Faster reads|
|Reason|Must check multiple SSTables|Data located in one place|

Reason for slower reads in LSM-tree:

```
Memtable → Multiple SSTables → Different compaction stages
```

---

# ==Benchmark Reality==

Performance comparisons are **workload dependent**.

Meaning:

```
Different workloads → different results
```

Therefore:

```
Always benchmark using your real workload
```

---

# Advantages of LSM-Trees

## 1. Higher Write Throughput

B-tree writes data multiple times:

```
1. Write to WAL
2. Write to tree page
3. Page split updates
```

LSM-tree writes:

```
Sequential SSTable writes
```

Sequential disk writes are **much faster than random writes**, especially on HDDs.

---

## 2. Lower Write Amplification (Sometimes)

**Write amplification**

Definition:

```
One logical write → multiple physical disk writes
```

Example sources:

- WAL
    
- Page splits
    
- Compaction
    

Lower amplification → higher throughput.

Important for:

```
SSDs (limited write cycles)
```

---

## 3. Better Compression

LSM-trees compress data more effectively because:

```
Sorted data + block compression
```

B-trees waste space due to:

```
Fragmentation
Half-filled pages after splits
```

Result:

|Structure|Storage Efficiency|
|---|---|
|LSM-tree|Smaller disk usage|
|B-tree|More fragmentation|

---

## 4. Sequential Disk Layout

LSM-tree writes **large sequential files**.

Benefit:

```
Better disk bandwidth usage
```

Especially beneficial for:

```
Magnetic hard drives
```

---

# ==Downsides of LSM-Trees==

## 1. Compaction Overhead

LSM-trees require **background compaction**.

Problem:

```
Compaction uses disk resources
```

Effects:

- Slower reads
    
- Slower writes
    

Sometimes causes:

```
High tail latency (slow worst-case queries)
```

B-trees often provide **more predictable latency**.

---

## 2. Compaction Falling Behind

If write rate is high:

```
Incoming writes > compaction speed
```

Then:

```
Unmerged SSTables increase
```

Consequences:

```
More disk usage
Slower reads
Possible disk exhaustion
```

Requires:

```
Monitoring and tuning
```

---

## 3. Multiple Copies of Keys

In LSM-trees:

```
Same key may exist in multiple SSTables
```

Because:

```
Older versions removed during compaction
```

In B-tree:

```
Each key stored in exactly one place
```

---

# ==Transactional Advantage of B-Trees==

Relational databases often use **range locks**.

Example:

```
Lock keys between A and B
```

This works naturally in B-trees because:

```
Keys stored in one ordered structure
```

LSM-trees make this harder.

---

# Practical Reality

|Property|LSM-Tree|B-Tree|
|---|---|---|
|Write-heavy workloads|Better|Worse|
|Read-heavy workloads|Worse|Better|
|Disk efficiency|Better|Worse|
|Latency predictability|Worse|Better|
|Transaction support|Harder|Easier|

# ==Other Indexing Structures== 

## 1. Primary Index vs Secondary Index

### Primary Index

- A **primary key** uniquely identifies a record in a database.
    
- Example:
    
    - Relational DB → row
        
    - Document DB → document
        
    - Graph DB → vertex
        
- Other records reference this record using the **primary key**.
    

Example:

```
User Table
ID (Primary Key) | Name
1                | Alice
2                | Bob
```

Primary index helps quickly find records using **ID**.

---

### Secondary Index

A **secondary index** allows searching using **non-primary fields**.

Example:

```
User Table
ID | Name | User_ID
```

You might create an index on **user_id** to quickly find all rows belonging to the same user.

In SQL:

```
CREATE INDEX idx_user_id ON table(user_id);
```

Important difference from primary index:

- **Primary index → unique key**
    
- **Secondary index → keys can repeat**
    

Example:

```
Key (user_id) → Row IDs
101 → [row1, row5, row8]
102 → [row2, row6]
```

Two ways to handle duplicate keys:

1. **Store list of row IDs**

```
key → [row_id1, row_id2, row_id3]
```

2. **Make key unique by appending row ID**

```
(user_id, row_id)
```

Both **B-Trees** and **LSM Trees** can implement secondary indexes.

---

# Storing Values Inside an Index

In an index:

- **Key → what we search**
    
- **Value → actual data OR reference to data**
    

Two main approaches exist.

---

## 1. Heap File (Most Common)

Index stores **pointer to the actual row**.

```
Index
Key → pointer to heap

Heap File
Actual records stored here
```

Heap file properties:

- Data stored **without order**
    
- Can be **append-only**
    
- Deleted rows can be reused
    

Example:

```
Index:
ID → address

Heap:
[address] → full record
```

### Advantages

- Avoids **data duplication**
    
- Multiple indexes can reference the same data
    

---

### Update Problem

If record size increases:

Old location may not have enough space.

Two solutions:

1️⃣ Move record and **update all indexes**

2️⃣ Leave **forwarding pointer**

```
Old location → pointer → new location
```

---

# Clustered Index

A **clustered index stores the actual data inside the index itself**.

```
Index Node
Key → Full Row Data
```

Example:

- MySQL InnoDB storage engine uses **clustered index for primary key**.
    

Secondary indexes store:

```
secondary key → primary key
```

instead of pointer to heap.

---

### Advantages

- Faster reads
    
- No extra lookup needed
    

### Disadvantages

- More storage
    
- Slower writes
    
- Data duplication issues
    

---

# Covering Index

A **covering index stores some columns of a table inside the index**.

Example:

Table:

```
id | name | age | city
```

Index might store:

```
(name, age)
```

If query only needs those columns:

```
SELECT name, age FROM users WHERE name='Alice'
```

Database **doesn't need to access table**.

This is called:

**Index covers the query**

---

### Tradeoff

Pros:

- Faster reads
    

Cons:

- More storage
    
- Extra write overhead
    

---

# Multi-Column Indexes

Sometimes queries filter using **multiple columns**.

Example:

```
SELECT * FROM users
WHERE last_name='Smith'
AND first_name='John'
```

---

## Concatenated Index

Combine multiple columns into **one key**.

Example:

```
(last_name, first_name)
```

Like a **phone book**:

```
(Smith, John) → phone number
```

Works for:

- last name only
    
- last + first name
    

Does NOT work for:

- first name only
    

Because order matters.

---

# Multi-Dimensional Indexes

Used when querying **multiple dimensions simultaneously**.

Common in:

- maps
    
- geospatial queries
    

Example:

Restaurant search:

```
SELECT *
FROM restaurants
WHERE latitude BETWEEN A AND B
AND longitude BETWEEN C AND D
```

A normal **B-Tree cannot efficiently handle this**.

Solutions:

### 1. Space-Filling Curve

Convert 2D coordinates → 1D number.

Then use normal index.

---

### 2. Spatial Index (R-Tree)

Special structure designed for multi-dimensional queries.

Used by:

- PostGIS
    
- PostgreSQL spatial indexing.
    

---

### Other Uses

Multi-dimensional indexes can be used for:

- **Color search**
    
    ```
    (red, green, blue)
    ```
    
- **Weather data**
    
    ```
    (date, temperature)
    ```
    

This allows filtering on **multiple dimensions simultaneously**.

---

# Full-Text Search & Fuzzy Indexes

Traditional indexes only support:

- Exact match
    
- Range queries
    

Example:

```
name = "apple"
```

They cannot handle:

- typos
    
- similar words
    
- synonyms
    

---

## Full-Text Search Engines

Provide features like:

- synonym matching
    
- grammatical variations
    
- proximity search
    
- typo tolerance
    

Example:

Search:

```
run
```

Matches:

```
running
ran
jog
```

---

Example engine:

- Apache Lucene
    

Lucene can search words within **edit distance**.

Edit distance = number of character changes.

Example:

```
book → books (1 change)
book → cook (1 change)
```

---

Lucene uses:

- **SSTable-like structure**
    
- **Trie-like automaton**
    
- **Levenshtein automaton** for fuzzy matching.

# In-Memory Databases

Most storage engines are optimized for **disk**.

But disk has problems:

- slow
    
- complex layout required
    

Advantages of disk:

1️⃣ Durable  
2️⃣ Cheap per GB

---

As **RAM becomes cheaper**, some databases keep everything in memory.

These are **in-memory databases**.

---

## Examples

Caching only (data loss acceptable):

- Memcached
    

Durable in-memory databases:

- Redis
    
- Couchbase
    
- VoltDB
    
- MemSQL
    
- Oracle TimesTen
    
- RAMCloud
    

---

## How Durability is Achieved

Even in-memory databases can persist data using:

1️⃣ **Write-ahead log on disk**  
2️⃣ **Periodic snapshots**  
3️⃣ **Replication to other machines**  
4️⃣ **Battery-powered RAM**

---

## Why They Are Faster

Not because they avoid disk reads.

Even disk databases often use **OS cache in RAM**.

Main advantage:

They avoid the overhead of:

```
encoding in-memory structures
→ into disk format
```

---

## Extra Benefits

In-memory databases can easily support complex structures:

Example in Redis:

- sets
    
- priority queues
    
- lists
    
- sorted sets
    

---

# Anti-Caching (Hybrid Approach)

If data becomes larger than RAM:

- Least Recently Used (LRU) data is moved to disk
    
- When needed → loaded back into memory
    

Similar to:

```
virtual memory + swap
```

But databases manage it **more efficiently than OS**.

---

# Future: Non-Volatile Memory (NVM)

New hardware types are emerging:

- persistent memory
    
- NVM technologies
    

These could change how **storage engines** are designed in the future.
